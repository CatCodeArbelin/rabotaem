# braceletbot

Телеграм-бот для оформления заказов на браслеты.

## Структура проекта

Структура в ТЗ трактуется как **ориентировочная**: в репозитории используется пакет `bot/` (без промежуточного каталога `project/`) и добавлены сервисные модули для разделения ответственности.

```text
.
├── bot/
│   ├── dialogs/                  # Диалоги, окна и состояния пользовательских сценариев
│   ├── keyboards/                # Клавиатуры и элементы интерфейса Telegram
│   ├── services/
│   │   ├── order_service.py      # Логика создания/сохранения заказов
│   │   └── notification_service.py # Отправка уведомлений администратору о новом заказе
│   ├── config.py                 # Загрузка и валидация настроек приложения
│   ├── database.py               # Инициализация и доступ к SQLite
│   ├── main.py                   # Точка входа и сборка зависимостей приложения
│   └── models.py                 # Доменные модели (например, заказ)
├── docker-compose.yml            # Локальный запуск и деплой в контейнере
├── Dockerfile                    # Сборка контейнера приложения
├── requirements.txt              # Python-зависимости
└── README.md
```

### Дополнительные модули

- `bot/services/notification_service.py` — отдельный сервис уведомлений, чтобы не смешивать бизнес-логику заказа и инфраструктурную отправку сообщений в админский чат.

## Локальный запуск (Windows + Docker Desktop)

1. Скопируйте файл переменных окружения:
   ```bash
   cp .env.example .env
   ```
2. Откройте `.env` и заполните обязательные переменные `BOT_TOKEN` и `PAYMENT_PROVIDER_TOKEN`.
   `PAYMENT_PROVIDER_TOKEN` нужно получить в BotFather в разделе **Payments** именно для того бота, чей токен указан в `BOT_TOKEN`.
3. Запустите проект:
   ```bash
   docker compose up --build
   ```

## Переменные окружения

Обязательные переменные:

- `BOT_TOKEN` — токен Telegram-бота из BotFather.
- `PAYMENT_PROVIDER_TOKEN` — provider token платежного провайдера Telegram Payments (например, ЮKassa), полученный в BotFather через раздел **Payments** для конкретного бота. Токен должен быть выпущен именно для бота из `BOT_TOKEN`; токен от другого бота не подойдет. Если переменная пустая, приложение завершится с ошибкой `Переменная окружения PAYMENT_PROVIDER_TOKEN не задана`.

> **Важно про режим оплаты:** если в платежном окне Telegram отображается кнопка `Оплатить (Test)`, используется тестовый provider token. Для приема реальных оплат нужен live-токен провайдера, выпущенный и подключенный в BotFather для этого же бота.

Необязательные переменные:

- `ADMIN_CHAT_ID` — ID чата администратора для уведомлений о новых заказах.

### Фискализация ЮKassa

По умолчанию бот **не передает** `provider_data.receipt` в `send_invoice`: это подходит магазинам, которые не фискализируют платежи через Telegram/ЮKassa или отправляют чеки другим способом. Чтобы ЮKassa сформировала чек из данных Telegram Payments, включите отправку receipt:

- `YOOKASSA_SEND_RECEIPT` — `true`/`false`; по умолчанию `false`. Установите `true` только если магазин подключил фискализацию через ЮKassa и готов передавать чек в `provider_data.receipt`.

При `YOOKASSA_SEND_RECEIPT=true` заполните обязательные данные для `provider_data.receipt` ЮKassa:

- `YOOKASSA_RECEIPT_EMAIL` или `YOOKASSA_RECEIPT_PHONE` — контакт покупателя для отправки чека; нужен хотя бы один из этих контактов. В инвойсе включены `need_email=True` и `send_email_to_provider=True`, но receipt все равно не должен быть пустым по контактам: если покупательский контакт нельзя корректно подставить заранее, оставьте `YOOKASSA_SEND_RECEIPT=false` для магазинов без фискализации через Telegram.
- `YOOKASSA_VAT_CODE` — ставка НДС для позиции в чеке; по умолчанию `1`.
- `YOOKASSA_PAYMENT_SUBJECT` — признак предмета расчета; по умолчанию `commodity`.
- `YOOKASSA_PAYMENT_MODE` — признак способа расчета; по умолчанию `full_payment`.
- `YOOKASSA_TAX_SYSTEM_CODE` — код системы налогообложения ЮKassa; заполните, если у магазина в ЮKassa настроено несколько систем налогообложения или ЮKassa требует этот параметр для вашей конфигурации.

Итоговая структура `provider_data.receipt` содержит объект `customer`, список `items` с описанием товара, количеством, суммой в RUB, `vat_code`, `payment_subject`, `payment_mode` и, если задано, `tax_system_code`.

## Troubleshooting платежей

### `PROVIDER_ACCOUNT_TIMEOUT`

Если Telegram возвращает ошибку `PROVIDER_ACCOUNT_TIMEOUT`, проверьте последовательно:

1. Провайдер платежей подключен в BotFather через раздел **Payments** именно к этому боту.
2. `PAYMENT_PROVIDER_TOKEN` в `.env` не пустой и не скопирован от другого бота.
3. Тестовый или боевой режим provider token соответствует ожидаемому режиму оплаты: кнопка `Оплатить (Test)` означает тестовый токен, для реальных оплат нужен live-токен.
4. Магазин/аккаунт ЮKassa активен, прошел необходимые проверки и не имеет ограничений на прием платежей.
5. Если настройки выглядят корректно, заново выпустите provider token в BotFather и обновите значение `PAYMENT_PROVIDER_TOKEN` в `.env`.

## Деплой на Ubuntu VDS

1. Клонируйте репозиторий:
   ```bash
   git clone <repo>
   ```
2. Перейдите в папку проекта:
   ```bash
   cd <repo>
   ```
3. Скопируйте шаблон переменных окружения:
   ```bash
   cp .env.example .env
   ```
4. Отредактируйте переменные окружения:
   ```bash
   nano .env
   ```
5. Запустите контейнер в фоне:
   ```bash
   docker compose up -d --build
   ```

## Эксплуатация

- Просмотр логов:
  ```bash
  docker compose logs -f
  ```
- Остановка контейнеров:
  ```bash
  docker compose down
  ```

## Сценарий заказа и хранение данных

Пользователь в боте выбирает товар из каталога, указывает параметры/детали заказа, оставляет контактные данные и подтверждает оформление. После подтверждения заказ сохраняется в SQLite-базе по пути:

`/app/data/orders.db` (в контейнере), что соответствует файлу `./data/orders.db` на хосте благодаря маунту `./data:/app/data`.
